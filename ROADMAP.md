# Codex Pet Club 迭代路线

本文按“先能安全上线，再提高可靠性，然后改善体验和增长”的顺序排列。

## 当前判断

系统已经完成产品闭环原型：用户能浏览真实桌宠动作，通过唯一 ID 调用 Skill 安装；创作者能通过 Skill 投稿；管理员能预览并审核。当前最大的差距不是页面设计，而是公开部署所需的身份、滥用防护、数据治理和可运维性。

## P0：上线硬门槛

### 1. 管理员认证与授权

- 保护 `/admin` 及全部 `/api/admin/*` 接口
- 使用服务端可信身份头或明确管理员 allowlist，不依赖前端隐藏入口
- 为审核动作记录管理员、时间、旧状态、新状态和备注
- 增加 CSRF/同源校验和安全响应头

验收标准：匿名用户无法读取待审稿件或执行审核；每次审核都有可追溯身份。

### 2. 投稿防滥用与资源保护

- 为 `POST /api/pets` 增加按用户/IP 的速率限制与每日配额
- 在解析 multipart 前限制请求体大小
- 对 ZIP 增加压缩比、单文件大小和内容类型检查
- 增加重复 SHA-256 检测、垃圾投稿拦截和失败清理任务
- 将作者与许可证改为必填并采用允许值列表

验收标准：单一来源无法耗尽 Worker、D1 或 R2 配额；重复和不合规投稿有明确错误。

### 3. 正式迁移与环境基线

- 停止在每个请求中执行建表、`PRAGMA` 和 `ALTER TABLE`
- 以版本化迁移作为部署前置步骤，修正当前 Drizzle migration 与 schema 的差异
- 明确开发、预发布、生产的 D1/R2 资源和配置
- 建立 D1 备份、R2 生命周期、回滚和恢复演练

验收标准：空环境可一键初始化；升级可重复执行；生产失败有可验证回滚路径。

### 4. 最小可发布流水线

- GitHub Actions 执行 lint、build、test
- 保护主分支，检查通过后才允许合并
- 先自动部署预发布环境，生产部署保留人工确认
- 增加健康检查、结构化错误日志和基础告警

验收标准：任一提交都能得到可复现的构建结果，部署状态与版本可追踪。

## P1：可靠性与规模化

### 5. API 与存储效率

- 公共目录改为服务端分页、搜索、排序和筛选
- 为预览图集生成小尺寸衍生资源，使用 ETag 与公共 CDN 缓存
- 避免每次预览都从 R2 读取并完整解压 ZIP
- 对 R2 对象和 D1 记录做定期一致性检查与孤儿清理
- 为 API 增加版本号、统一错误码和请求 ID

### 6. 真正的 Skill 通道可信度

- 明确目标：若只需阻止普通网页下载，当前自定义请求头足够表达约定；若要访问控制，则必须改用短期签名 URL 或服务端签名令牌
- 为包下载记录版本、校验和、客户端版本与失败原因
- 支持包撤回、禁用、替换和 Skill 端安全回滚

### 7. 自动化测试补齐

- 为上传校验器增加单元测试：路径穿越、ZIP bomb、清单错误、尺寸错误、重复文件
- 使用本地 D1/R2 跑 API 集成测试
- 覆盖投稿 → 审核 → 公开展示 → Skill 下载的 E2E 主链路
- 增加并发审核、重复 slug、对象缺失和迁移兼容测试

## P2：产品体验

### 8. 可分享的桌宠详情页

- 增加 `/pets/:id` 独立页面和稳定 URL
- 支持动作切换、暂停、键盘操作、加载状态和分享元信息
- 显示作者链接、许可证说明、版本和更新时间

### 9. 桌宠发现能力

- 分类、标签、风格、排序、分页与热门/最新筛选
- 收藏、安装量与精选机制，但避免用可伪造数据制造热度
- 为空目录、审核中、下架和版本不兼容提供清晰状态

### 10. 创作者与审核体验

- 创作者查看投稿状态、失败原因并重新提交
- 审核台支持差异对比、批量操作、风险提示和历史记录
- 桌宠版本化，允许保留旧版、灰度发布和回滚

## P3：生态与增长

- 作者主页、合集、桌宠专题与社区推荐
- Skill 自动更新、兼容性声明和客户端版本协商
- 内容举报、版权申诉、下架与恢复流程
- 在数据量足够后再引入质量评分或自动审核辅助

## 推荐执行顺序

1. 管理员认证与审核审计
2. 投稿限流、配额、ZIP 防护与许可证约束
3. 数据库迁移、环境隔离、备份与回滚
4. CI、预发布部署、健康检查和日志告警
5. 预览资源拆分与缓存、API 分页和对象一致性
6. Skill 下载签名、撤回与版本回滚
7. D1/R2 集成测试和全链路 E2E
8. 独立桌宠详情页与可访问性
9. 搜索筛选、创作者中心和审核效率
10. 社区增长、举报申诉和自动审核辅助

前四项完成后再公开部署；前七项完成后，系统才具备稳定扩大桌宠数量和用户量的基础。
